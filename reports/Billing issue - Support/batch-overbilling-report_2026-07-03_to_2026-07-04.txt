PURPOSE
-------
This document records, with evidence pulled directly from OpenAI's own
Administration (usage & costs) API, a series of Batch API runs in which the
account was charged for a large volume of model executions that produced NO
returned output. It is written so it can be adapted into a refund request. It
is a factual evidence record, not the request itself.

All figures below are taken from:
  - GET /v1/batches/{id}                          (per-batch request_counts + usage)
  - GET /v1/organization/usage/completions        (billed tokens & model-request counts)
  - GET /v1/organization/costs                     (authoritative dollar charges)
and cross-checked against the batch output/error files.

2. EXECUTIVE SUMMARY (THE CLAIM IN BRIEF)
--------------------------------------------------------------------------------
Across three Batch API jobs in this window:

  - Requests SUBMITTED by us .................... 43
  - Requests that COMPLETED (returned output) ... 8
  - MODEL EXECUTIONS actually billed ............ 89   (2.1x the requests submitted)
  - Batch API dollars billed ................... $43.58
  - Value of the 8 completed requests ........... $2.38
  => Billed for work that produced NO output ... ~$41.20

OpenAI's stated Batch API billing policy (OpenAI support, OpenAI Developer
Community, thread "Batch API billing for failed requests (504 Timeout)"):
  "billing is based on token usage from completed requests"; failed requests
  that appear in the error file "without response or usage data are excluded
  from billing."

The charges below are inconsistent with that policy: the account was billed for
4.81M input and 2.14M output tokens, while the 8 requests that actually returned
output account for only 0.24M input and 0.12M output tokens. The remaining
~4.57M input / ~2.02M output tokens were consumed by requests that FAILED and by
undisclosed internal retries of those failed requests, and returned nothing.

4. KEY EVIDENCE
--------------------------------------------------------------------------------
(A) EXECUTIONS FAR EXCEED REQUESTS SUBMITTED (undisclosed internal retries)
      Requests submitted (all 3 batches) .... 43
      Model executions billed ............... 89
      Ratio ................................. 2.07x   (Batch-3 alone: 27 / 6 = 4.5x)
    The Batch API documentation exposes no retry parameter and does not disclose
    that failed requests are retried. The only explanation for 89 executions of
    43 requests is undisclosed server-side retries, each one billed.

(B) BILLED TOKENS FAR EXCEED COMPLETED-REQUEST TOKENS
                              INPUT tokens      OUTPUT tokens
      Billed (usage API) .... 4,810,600         2,136,377
      Completed requests .... 236,227           119,223
      Billed for nothing .... 4,574,373         2,017,154
    ~95% of the input tokens and ~94% of the output tokens billed produced no
    returned result.

(C) DOLLARS: BILLED vs. VALUE DELIVERED
      Batch API billed (costs API) ............ $43.58
      Value of the 8 completed requests ....... $2.38
      Charged for zero-output work ............ ~$41.20

(D) FAILURES WERE NOT CAUSED BY MALFORMED INPUT ON OUR SIDE
      - HTTP 503 "server_is_overloaded" is explicitly a server-side condition.
      - HTTP 400 "There was an issue with your request" was returned with NO
        error code and a generic message. The same 5 image files that triggered
        it are identical in format (PNG), dimensions (~4050 x 6800 px) and size
        range to files that SUCCEEDED in the same batches, so the input itself
        was valid and accepted.
      - HTTP 429 insufficient_quota in Batch-1/2 was itself a consequence of the
        retry storm draining the balance mid-run; each retried execution was
        billed before the balance check failed.

    DIRECT PROOF THE 400 IS A FALSE (SERVER-SIDE) ERROR:
      Scan FN-0006 failed with HTTP 400 "There was an issue with your request.
      Please check your inputs and try again" in the Batch API (in both Batch-1
      and Batch-3). The SAME file was then submitted via the SYNCHRONOUS
      /v1/responses API with the IDENTICAL configuration (gpt-5.5, effort high,
      Code Interpreter on, same prompt) on 2026-07-04 07:45 UTC. It COMPLETED
      SUCCESSFULLY on the first attempt: 39,807 input / 19,851 output tokens,
      returning a full transcription for ~$0.79. The input was therefore valid
      and processable; the Batch API's 400 "check your inputs" response was a
      false, server-side failure -- and it was that false failure, retried
      internally, that generated the bulk of the zero-output charges.
